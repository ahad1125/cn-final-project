Network Fingerprint Generator & Website Behavior 
Profiler 
 
1. Project Overview 
This project builds a web-based tool that captures live network traffic when a user visits a website, analyzes the 
captured packets, and produces a unique network fingerprint. A fingerprint is a compact behavioral profile 
summarizing how the site communica tes: which protocols it uses, how large its packets are, how frequently it 
sends data, and what servers it contacts. 
The tool is designed as an educational aid for networking students. By comparing fingerprints of different websites, 
students can visually observe real -world differences in network behavior, for example a video -streaming site 
versus a static news page. 
2. Detailed Objectives 
2.1 Capture Network Traffic 
Use Scapy to sniff packets at the network interface while a target website is accessed. Capture is scoped to traffic 
from the entered URL only, avoiding noise from background processes. 
2.2 Extract Meaningful Features 
From captured packets, extract: packet sizes in bytes, protocol distribution (TCP , UDP , DNS, HTTPS), inter-arrival 
times, destination IP and port pairs, and total data volume. 
2.3 Generate a Network Fingerprint 
Aggregate extracted features into a structured summary including statistics like mean and max packet size, 
dominant protocol, number of unique hosts contacted, and a normalized traffic vector. 
2.4 Classify Website Behavior 
Apply rule-based to label site behavior such as Streaming, Social Media, Static Content, or API -heavy, helping 
students understand traffic without reading raw dumps. 
2.5 Enable Side-by-Side Comparison 
Allow the user to enter two URLs. The system captures both, generates fingerprints, and presents a comparison 
highlighting differences in protocols, data volume, and request frequency. 
2.6 Visualize Traffic Characteristics 
Present results through Chart.js graphs: protocol pie charts, packet -size histograms, and traffic -over-time line 
graphs rendered dynamically in the browser. 
3. Functional Requirements  
 
# Requirement What It Means in Practice 
FR-1 Website URL Input 
A clearly labeled text field on the frontend accepts a full URL (e.g., https://example.com). 
Before the URL is sent to the backend, JavaScript validation checks that it starts with http:// 
or https:// and has a valid domain format. If the format is wrong, an inline error message is 
shown immediately without making a server request. Once valid, the URL is submitted via a 
POST request to the Flask backend.  
FR-2 Packet Capture 
The Flask backend triggers a Scapy packet sniffer on the active network interface for a 
configurable capture window (default: 10 seconds). Simultaneously, an automated 
HTTP/HTTPS request is made to the target URL using Python requests or a headless 
browser to generate real traffic. All captured packets are stored in a temporary .pcap file on 
disk for later processing. The sniffer runs in a separate thread so Flask remains responsive 
during capture. 
FR-3 Feature Extraction 
After capture completes, a dedicated extract.py module reads the .pcap file using Scapy's 
rdpcap() function and iterates over every packet to compute: (1) total packet count, (2) 
protocol breakdown as percentages (TCP, UDP, DNS, ICMP, ARP, HTTPS), (3) individual 
packet sizes in bytes, (4) mean, minimum, and maximum packet size, (5) list of unique 
destination IP addresses contacted, (6) DNS query names made during the session, (7) total 
bytes transferred, and (8) packet inter -arrival times. All values are stored in a Python 
dictionary passed to the fingerprint module.  
FR-4 Fingerprint 
Generation The fingerprint.py module receives the extracted feature dictionary and normalizes values 
where needed (e.g., protocol percentages sum to 100%). It assembles a structured JSON 
fingerprint object with clearly labeled fields: site_url, capture_timestamp, tot al_packets, 
total_bytes, top_protocol, unique_ips, dns_queries, mean_packet_size, max_packet_size, 
protocol_distribution, and behavior_label. This JSON is the core output of the system and is 
passed both to the frontend for display and to the classifier fo r labeling. 
FR-5 Traffic Classification 
The classify.py module reads the fingerprint JSON and applies a simple rule -based classifier 
to assign one of the following behavior labels: (1) Streaming - high byte volume, large 
average packet size, mostly TCP; (2) Social Media - frequent small packets,  many unique 
IPs, mix of protocols; (3) Static Content - low packet count, short session duration, minimal 
DNS queries; (4) API-Heavy - very small packets, rapid request -response cycles, HTTPS 
dominant; (5) Unknown - does not match any pattern. The label a nd a confidence percentage 
are included in the fingerprint JSON returned to the frontend.  
FR-6 Website Comparison 
The frontend provides a second URL input field for comparison mode. When both URLs are 
submitted, the backend performs two sequential captures and generates two separate 
fingerprints. A comparison endpoint (/api/compare) returns both fingerprints in a sing le JSON 
response along with a diff object highlighting key differences: which site used more total 
bytes, which had more unique IPs, which had higher mean packet size, and which behavior 
label each received. The frontend renders both fingerprint cards side  by side with color-
coded difference indicators (green for lower, red for higher).  
FR-7 Visualization 
All charts are rendered client-side using Chart.js after the fingerprint JSON arrives from the 
backend. Three charts are required: (1) Protocol Pie Chart - shows percentage share of each 
protocol detected in the capture session; (2) Packet Size Histogram - groups packets into 
size buckets (0-100 bytes, 101-500 bytes, 501-1000 bytes, 1001-1500 bytes) and shows 
count per bucket as a bar chart; (3) Traffic Timeline - plots total bytes transferred per second 
over the capture window as a line chart. In compariso n mode, the timeline shows two 
overlapping lines (one per website) in different colors for easy contrast.  
4. Non-Functional Requirements Explained 
NFR-1 User-Friendly Interface 
The UI guides the user step by step: enter URL, start capture, view results. Error messages must be human -
readable, for example: Could not reach the website, check your URL. 
NFR-2 Fast Processing 
Capture window defaults to 10 seconds. Feature extraction and fingerprint generation should complete within 3 
seconds after capture ends. Total workflow from URL entry to results: under 20 seconds. 
NFR-3 Dataset Size Handling 
The system handles pcap files up to approximately 5 MB comfortably using in-memory parsing. No streaming or 
chunked processing is required for this academic scope. 
NFR-4 Modular Codebase 
Code is split into clear modules: capture.py, extract.py, fingerprint.py, classify.py, and app.py for Flask routes. 
Each module should be independently testable. 
5. Technology Stack and Role of Each Component 
Technology Role in the Project 
Python 3.x Core programming language for all backend logic and data processing. 
Scapy Low-level packet capture and parsing. Reads raw frames off the network interface. 
Flask Lightweight web server exposing REST API endpoints the frontend calls via AJAX. 
HTML / CSS Frontend structure and styling. A clean single-page layout for input and results. 
JavaScript Handles API calls, processes JSON responses, and triggers chart rendering. 
Chart.js Renders all visualizations including pie, bar, and line charts in the browser canvas. 
pcap / JSON Intermediate data formats: pcap for raw packet storage, JSON for fingerprint exchange. 
6. Expected Outputs 
Output 1: Network Fingerprint Summary Card 
A structured JSON object displayed as a UI card containing: site URL, capture timestamp, total packets, total 
bytes, top protocol, unique destination IPs, DNS queries made, mean packet size, and behavior classification 
label. 
Output 2: Protocol Distribution Graph 
A pie or doughnut chart showing what percentage of packets used TCP, UDP , DNS, ICMP, etc. Helps students see 
that HTTPS over TCP dominates most modern websites. 
Output 3: Packet Size Histogram 
A bar chart grouping packets by size range. Streaming sites produce large packets while API-heavy sites produce 
many small ones. 
Output 4: Traffic Timeline 
A line chart plotting packet count or byte volume per second over the capture window. Reveals burst patterns, 
connection setup spikes, and idle periods. 
Output 5: Comparison Report 
A side-by-side table and overlay charts for two websites showing which metrics differ and by how much. 
8. System Workflow Step by Step 
Step 1 - URL Input 
User opens the web interface, enters a target URL, and clicks Analyze. 
Step 2 - Backend Trigger 
Frontend sends a POST request to Flask endpoint /api/analyze with the URL as payload. Step 
3 - DNS Resolution 
Flask resolves the URL to IP addresses using Python's socket library to filter packets later. Step 
4 - Packet Capture 
Scapy sniffs the active interface for the configured duration while a request to the URL generates traffic. 
Step 5 - Feature Extraction capture ends and extract.py processes the packet list to compute all 
statistics.  
Step 6 - Fingerprint Assembly 
fingerprint.py packages statistics into JSON and calls classify.py to add the behavior label. 
 Step 7 - API Response 
Flask returns the JSON fingerprint to the frontend. 
Step 8 - Visualization 
JavaScript receives JSON, populates the summary card, and Chart.js renders all graphs.
